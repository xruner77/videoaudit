### 高危 #1 — 密钥硬编码于配置文件

> **位置:** [db_config.php](file:///d:/tfx/myapp/videoaudit/server/db_config.php)，第 7、9 行  
> **维度:** 安全性  
> **复核状态:** ⚠️ 降级（原严重→高危）

**问题:** JWT 签名密钥（`va_jwt_s3cr3t_k3y_2026_xR9mNpL7qW`）和 MySQL 凭据（`videoaudit / videoaudit123`）被硬编码在配置文件中。`.gitignore` 已正确排除该文件，**未泄露至 Git 历史**（已验证）。但硬编码密钥的做法仍存在风险：如果有人 fork 仓库时手动添加配置、或者服务器被入侵读取文件，密钥即暴露。最佳实践是使用环境变量管理敏感信息。

**修复方案:**
1. 将密钥迁移至 **环境变量**（如 `$_ENV['JWT_SECRET']`）或使用 `vlucas/phpdotenv` 加载 `.env` 文件。
2. 在生产服务器上定期轮换 JWT 密钥和数据库密码。
3. ~~使用 `git filter-repo` 清除历史~~ — **不需要**（经复核确认文件从未入库）。

```php
// 修复前 (db_config.php):
'jwt_secret' => 'va_jwt_s3cr3t_k3y_2026_xR9mNpL7qW',

// 修复后 (db_config.php):
'jwt_secret' => getenv('JWT_SECRET') ?: throw new \RuntimeException('JWT_SECRET 未设置'),
```

### 高危 #2 — 评论接口 IDOR 越权访问

> **位置:** [comments.php](file:///d:/tfx/myapp/videoaudit/server/src/Routes/comments.php)，第 80 行  
> **维度:** 安全性  
> **复核状态:** ✅ 已确认

**问题:** `GET /api/comments/user/{userId}` 接受 **任意** `userId` 路径参数。任何已认证用户都可以通过遍历用户 ID 来枚举和读取其他用户的全部评论。该路由仅要求登录（`AuthMiddleware` 未开启 `adminOnly`），但 **不验证** 调用者的 JWT `sub` 是否匹配请求的 `userId`。

**修复方案:** 强制所有权校验 — 要么限制 `userId` 必须匹配 JWT 中的 `sub`，要么移除路径参数，始终从 token 中获取用户 ID：

```php
// 修复前:
$userId = (int) $args['userId'];

// 修复后（方案 A：强制所有权）:
$caller = $request->getAttribute('user');
$userId = (int) $args['userId'];
if ($userId !== (int) $caller->sub) {
    $response->getBody()->write(json_encode(['error' => '无权访问']));
    return $response->withStatus(403)->withHeader('Content-Type', 'application/json');
}
```

### 高危 #3 — 敏感接口无速率限制

> **位置:** [index.php](file:///d:/tfx/myapp/videoaudit/server/public/index.php)（全局），[auth.php](file:///d:/tfx/myapp/videoaudit/server/src/Routes/auth.php) 第 19 行  
> **维度:** 生产风险 / 安全性  
> **复核状态:** ✅ 已确认

**问题:** 登录接口、文件上传接口和评论创建接口 **没有任何速率限制**。攻击者可以无限制地暴力破解密码或恶意灌水上传 / 评论。

**修复方案:** 添加兼容 Slim 的速率限制中间件（如基于 IP 的令牌桶，存储在 Redis 或文件缓存中）。最低优先级应用于：
- `POST /api/auth/login` — 每 IP 每分钟最多 10 次
- `POST /api/videos/upload` — 每用户每小时最多 5 次
- `POST /api/comments` — 每用户每分钟最多 30 次

### 高危 #4 — 管理员用户列表关联子查询性能问题

> **位置:** [auth.php](file:///d:/tfx/myapp/videoaudit/server/src/Routes/auth.php)，第 173–181 行  
> **维度:** 生产风险  
> **复核状态:** ✅ 已确认

**问题:** `GET /api/admin/users` 对用户表的 **每一行** 都执行两个关联子查询（`SELECT COUNT(*) FROM videos WHERE user_id = u.id` 和 `SELECT COUNT(*) FROM comments WHERE user_id = u.id`）。这本质上是 N+1 查询行为 — 100 个用户勉强可用，但随用户数增长性能线性下降，且无分页机制。

**修复方案:** 使用 `LEFT JOIN ... GROUP BY` 替代关联子查询，并增加分页：

```sql
SELECT u.id, u.username, u.role, u.created_at,
       COALESCE(v.cnt, 0) AS video_count,
       COALESCE(c.cnt, 0) AS comment_count
FROM users u
LEFT JOIN (SELECT user_id, COUNT(*) cnt FROM videos GROUP BY user_id) v ON v.user_id = u.id
LEFT JOIN (SELECT user_id, COUNT(*) cnt FROM comments GROUP BY user_id) c ON c.user_id = u.id
ORDER BY u.created_at DESC
LIMIT ? OFFSET ?
```

### 中等 #3 — 退出登录逻辑重复

> **位置:** [profile.vue:176–190](file:///d:/tfx/myapp/videoaudit/src/pages/profile/profile.vue#L176-L190) vs [authStore.js:48–54](file:///d:/tfx/myapp/videoaudit/src/stores/authStore.js#L48-L54)  
> **维度:** 一致性与可维护性  
> **复核状态:** ✅ 已确认

**问题:** `profile.vue` 手动清除 `token`、`user` 和 storage 存储项，而不是调用 `authStore.logout()`。如果退出登录流程未来需要清除额外状态（如推送 token），`profile.vue` 会被遗漏。

**修复方案:** 将 `profile.vue` 中的内联退出逻辑替换为 `authStore.logout()`：

```js
// 修复前 (profile.vue):
authStore.token = ''
authStore.user = null
uni.removeStorageSync('token')
uni.removeStorageSync('user')
uni.reLaunch({ url: '/pages/login/login' })

// 修复后:
authStore.logout()
```
