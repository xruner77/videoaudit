## 高风险问题 (HIGH)

### [高风险 #1] 缺少 /api/videos/{id} 接口 -- 审片页拉取了全量视频列表

- **位置:** src/pages/review/review.vue, 第 236-267 行
- **维度:** 生产风险 / 架构设计
- **已确认:** review.vue 第 239 行调用 GET /api/videos, 第 243 行使用 .find() 客户端过滤

**问题:** fetchVideoDetail() 调用了 GET /api/videos (该接口返回带分页的所有视频), 然后通过 .find(v => v.id == videoId.value) 在前端查找。当视频数量超过 100 个时, 会产生明显的延迟和流量浪费。

**修复建议:** 在后端增加 GET /api/videos/{id} 接口:

```php
// 修改前 (review.vue 获取所有视频):
const res = await request({ url: `/api/videos`, method: 'GET' })
const video = (res.data.videos || []).find(v => v.id == videoId.value)

// 修改后 (仅获取单个视频):
const res = await request({ url: `/api/videos/${videoId.value}`, method: 'GET' })
const video = res.data.video
```

### [高风险 #2] 级联删除缺少事务保护 -- 数据损坏风险

- **位置:** server/src/Routes/auth.php, 第 263-266 行
- **维度:** 健壮性
- **已确认:** 第 264-266 行连续执行 3 条 DELETE, 无 beginTransaction/commit

**问题:** 管理员删除用户时, 连续执行 3 条 DELETE 语句 (评论 -> 视频 -> 用户) 但没有开启事务。如果中间某条语句失败 (如外键约束、超时), 数据库会残留孤立记录。

**修复建议:** 使用事务包裹:

```php
$db->beginTransaction();
try {
    $db->prepare('DELETE FROM comments WHERE user_id = ?')->execute([$targetId]);
    $db->prepare('DELETE FROM videos WHERE user_id = ?')->execute([$targetId]);
    $db->prepare('DELETE FROM users WHERE id = ?')->execute([$targetId]);
    $db->commit();
} catch (\Exception $e) {
    $db->rollBack();
    throw $e;
}
```

### [高风险 #3] 文件上传过度信任客户端 MIME 类型

- **位置:** server/src/Routes/videos.php 第 123 行; comments.php 第 182 行
- **维度:** 安全性
- **已确认:** videos.php 第 123 行使用 $file->getClientMediaType(), 仅校验客户端 Header

**问题:** 视频和图片上传接口使用 $file->getClientMediaType() 验证类型, 这读取的是客户端发送的 Header, 极易被伪造。攻击者可以上传改了后缀名的 PHP 脚本。

**修复建议:** 使用 finfo 在服务端验证真实 MIME 类型:

```php
$finfo = new \finfo(FILEINFO_MIME_TYPE);
$realMime = $finfo->file($filePath);
if (!in_array($realMime, $allowedTypes)) {
    unlink($filePath);
    // return 400 error
}
```

### [高风险 #4] CORS 中间件对未知来源的回退逻辑漏洞

- **位置:** server/public/index.php, 第 33 行
- **维度:** 安全性
- **已确认:** 第 33 行代码: $originToSet = in_array($origin, $allowedOrigins) ? $origin : ($allowedOrigins[0] ?? '*');

**问题:** 当请求来源不在白名单时, 代码会默认回退到白名单第一个域名。这意味着任何请求都会得到一个合法的 CORS 响应头, 虽然不会直接导致跨站访问, 但违反了 CORS 安全模型。

**修复建议:** 对不在白名单的 Origin, 不返回任何 CORS 相关响应头:

```php
// 修改前:
$originToSet = in_array($origin, $allowedOrigins) ? $origin : ($allowedOrigins[0] ?? '*');

// 修改后:
if (!in_array($origin, $allowedOrigins)) {
    return $handler->handle($request); // 不添加 CORS 头
}
$originToSet = $origin;
```

### [高风险 #5] 缺少登录频率限制 -- 暴力破解风险

- **位置:** server/src/Routes/auth.php, 第 19-57 行
- **维度:** 安全性
- **已确认:** login 路由 (第 19-57 行) 无任何限流逻辑, 直接查库校验

**问题:** POST /api/auth/login 接口没有任何限流措施。攻击者可以每秒发起成百上千次尝试, 配合 6 位数的短密码要求, 爆破成功率较高。

**修复建议:** 在 Nginx 层添加频率限制:

```nginx
# nginx.conf
limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;
location = /api/auth/login {
    limit_req zone=login burst=3 nodelay;
    proxy_pass http://127.0.0.1:8080;
}
```
